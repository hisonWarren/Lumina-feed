# 文献发现 + 每日 AI 推送 · 多角色评议总卷

> **产品代号（暂定）**：**Lumina Feed**（文献雷达 / 每日证据简报）
> **一句话**：一个本地优先的独立程序——按你的研究主题持续检索多个学术库，**每天**把当天新发表、命中你需求的文献，连同 AI 摘要（能取全文就基于全文，取不到就基于标题摘要，且这一切由你设选项）推送给你；平时则是一个**文献数据库风格**的可筛选探索台（年份、期刊、类型……）。
> **流程**：本卷按你指定的**会议流水线**推进——**会议 0（流程/形态选型）→ 会议 1（用户需求）→ 会议 2（方案）→ 会议 3（实现蓝图 + 原型）**，每场会议消费上一场的产出（文末有可追溯矩阵）。
> **签发**：2026-06-26

## 全局红线（贯穿所有会议，非协商）

延续你既有工程文化（`ADR-SRCH-05` / `OA_DENY_PATTERNS` / `docs/05_合法性与限制.md`）：

1. **自动下载 = 仅合法开放获取（OA）**。Sci-Hub / LibGen / Anna's / 任何绕付费墙的爬虫**一行不写**。取不到全文 = 诚实降级（机构访问链接 / 基于摘要总结），不碰灰色源。
2. **守各 API 的 ToS 与速率/署名要求**：NCBI E-utilities（带 `tool`+`email`，无 key ≤3 req/s、有 key ≤10）、Crossref/OpenAlex 走 polite pool（带 mailto）、arXiv/bioRxiv 温柔轮询、不抓出版商 HTML 正文。本地缓存 + 指数退避。
3. **AI 只排序与总结，绝不替你做"纳入/排除"决策**；每条摘要标清**依据（全文 / 仅摘要）**并附原文链接；预印本、撤稿、利益冲突等信号显式标注。
4. **隐私与密钥用户自持**：LLM 走你自己的 key（Claude / OpenAI / 本地 Ollama 均可），密钥存系统钥匙串，绝不明文落库；支持**全本地**总结路径。

**与会**：R6 架构 Mei、R10 YAGNI Sol、R11 合规 Audit、R-DataEng 数据工程、R-Sec 安全、R-PM 产品协调。

## 0.1 先校准流程本身

你给的线性流水线（需求→方案→代码）**基本正确，但补两点**：

- **加一场"会议 0"**（本场）：因为"每日推送"在不同部署形态下可行性天差地别，**不先定形态，需求会会架空**。Sol：「别让需求会去许诺一个桌面程序睡眠时还能推送的承诺。」
- **流水线不是单向瀑布，是带回环的**：会议 2 若发现某需求在选定形态下**工程不可行/代价过高**，回写会议 1 标注「降级/延后」，而非硬塞。Mei：「需求是输入，不是合同。」

✅ **结论**：采用 **0→1→2→3 四段 + 回环**。本卷一次性走完四段（每段产出喂下一段），并在文末给可追溯矩阵。

## 0.2 形态/部署三叉（决定一切的 crux）

「每日推送」的硬真相：**一个不常开机的桌面程序，无法可靠地在每天早 8 点推送。** 三条路：

| 形态 | 每日推送如何成立 | 优点 | 代价 / 真相 |
|---|---|---|---|
| **A. 纯本地桌面（开机即追赶）** | App 启动 / 开机自启时，拉取「上次运行至今」的新文献，生成当日简报 | 零服务器、零成本、隐私满分、最简单 | **只在你打开 App 时才有简报**；睡眠期不推送（醒来补） |
| **B. 本地桌面 + 常驻后台 sidecar** | 一个无界面的小后台（launchd / 计划任务 / systemd 或 Tauri sidecar）按时跑检索 → 发**系统通知** + 写库 | 真·定时、仍全本地、可弹原生通知 | 需装后台进程、跨 OS 适配；机器关机仍不推 |
| **C. 桌面 + 极小云 worker** | 云端 cron（小 VPS / 云函数）跑检索+总结，经**邮件 / Telegram / Webhook** 推到手机；桌面同步 | 真·24/7、推到手机、机器关机也行 | 引入服务器（成本 + 运维）；总结在云上跑（key/成本/隐私需权衡） |

**R-Sec**：C 的云端要放 LLM key 和你的检索主题——除非自托管，否则隐私让步。
**R-DataEng**：检索 API 多数无需鉴权，云端跑检索没问题；**总结才是成本/隐私大头**。
**Audit**：A/B 全本地最稳合规；C 需明确数据留存与第三方处理告知。

### ✅ 决选：**A 为基线，B 为可选增强，C 为可选"推到手机"通道**（渐进、不强制）

- **MVP = A**：本地优先桌面，开机/启动追赶式简报 + 应用内"今日推送"视图 + 原生通知（App 在跑时）。**零基础设施即可用**。
- **增强 = B**：愿意的用户一键开启"开机自启 + 后台定时"，获得真·定时本地通知。
- **可选 = C**：想推到手机的用户，填自己的邮箱/Telegram/Webhook；推送 worker 可复用同一套逻辑，**在你常开的机器上**或**自托管小服务**运行（默认不提供托管云，避免替用户承担 key/成本/合规）。
- **诚实文案**（写进 UI）：「若本机休眠，你会在下次唤醒时收到补发的简报；要真正 24/7 推到手机，需一台常开的机器或自建小服务。」

## 0.3 技术栈选型（给会议 2/3 定锚）

| 维度 | 选择 | 理由 |
|---|---|---|
| 外壳 | **Electron + better-sqlite3**（主荐）/ Tauri（更轻，Rust，备选） | 你已有 TS/React/Electron 栈，迁移成本最低；Tauri 体积/内存更优但需 Rust |
| 前端 | **React + TypeScript + Tailwind** | 与你现有 Synra 一致，复用经验 |
| 存储 | **SQLite（本地单文件）+ FTS5 全文索引** | 文献库天然关系型；FTS5 给标题/摘要全文检索，普通索引给 facet 列 |
| 检索 | 各库官方 API 适配器（见会议 2） | 合法、稳定、CORS/ToS 友好 |
| 调度 | `node-cron`（主进程）+ `auto-launch`（开机自启）+ 可选 sidecar | A/B 形态落地 |
| 摘要 | **可插拔 LLM 客户端**（Anthropic / OpenAI / 本地 Ollama），key 入系统钥匙串 | 用户自持、可全本地 |
| 全文 | `pdfjs-dist` 抽文本（轻）/ GROBID（可选，结构化） | 复用你 OA 取全文红线逻辑 |
| 推送 | 原生通知 / 邮件(SMTP) / Telegram bot / 通用 Webhook | A/B/C 三形态通道 |

# 会议 1 · 用户需求会（多领域用户角色）

**与会用户角色（7 位，跨领域）**，每位给：画像 · 典型场景 · 核心诉求 · 痛点 · **对界面与内容的具体期待** · 触及红线。
主持人 R-PM 负责把发散诉求收敛成台账。

### U1 · 临床医生（循证实践 / 三甲心内科主治）
- **场景**：每天清晨 10 分钟，想知道「我管的病种有没有可能改变我处置的新 RCT / 指南更新」。手机为主。
- **核心诉求**：高信噪比、**先给"是否值得我读 / 是否改变实践"判断**、按期刊分级（JCR 分区 / IF）与研究类型（RCT > 观察）过滤、移动端晨读。
- **痛点**：PubMed 结果噪声大、预印本误当定论、没时间读全文。
- **对界面/内容**：**"今日推送"卡片式**，每条一句话 TL;DR + "临床意义"标签（改变实践 / 仅供参考 / 阴性结果）；按 **研究类型 + 期刊分级**强筛；显示样本量、是否多中心。
- **红线**：要 AI 帮**判断重要性**，但**不要替我下临床结论**——给依据让我判断。

### U2 · 基础科研 PhD（窄领域 / 神经免疫）
- **场景**：紧盯一个很窄的方向，预印本和正式发表都要第一时间看到；看到相关的就要全文进 Zotero。
- **核心诉求**：**preprint（bioRxiv/arXiv）+ 正式发表都覆盖**、关键词/作者/实验室订阅、**一键取全文 PDF**、方法学细节、与已读去重。
- **痛点**：同一篇 preprint→正式发表算两条很烦、跨库重复、全文获取麻烦。
- **对界面/内容**：**数据库式长列表 + 强力 facet**（年份滑块、来源、preprint/peer-reviewed、物种/方法 MeSH）；**preprint↔published 版本归并**显示；导出 BibTeX / RIS / 直推 Zotero。
- **红线**：取全文只走合法 OA；取不到给机构链接。

### U3 · 系统综述方法学家（Cochrane 风 / 循证医学中心）
- **场景**：做系统综述，需要**全面、无偏、可复现**的检索，并持续监控有无新研究需纳入更新。
- **核心诉求**：**结构化可复现检索式**（布尔 + 字段 + MeSH 爆炸）、跨库去重、PRISMA 计数、**检索式可导出存档**、新命中提醒（用于活体综述 living review）。
- **痛点**：怕漏检（敏感度）、怕 AI 偷偷帮我排除、检索过程不可复现没法写进方法学。
- **对界面/内容**：**可视化检索构造器** + 原始检索式文本双向同步；**导出"检索策略 + 命中数 + 日期"审计包**；命中流入"待筛"而非自动判定。
- **红线（一票否决级）**：**AI 绝不能替我做纳入/排除**；只能排序/打标，决策权 100% 在我；全过程留痕。

### U4 · 药企竞争情报分析师
- **场景**：监控竞品靶点 / 适应症 / 临床阶段的新文献与会议摘要，定期出简报给团队。
- **核心诉求**：按**申办方 / 靶点 / 适应症 / 试验阶段**监控、趋势统计、**导出可分享报告**、数据可留存合规。
- **痛点**：信息分散、要人工汇编报告、合规要求严（版权 / 数据出境）。
- **对界面/内容**：**多订阅并行的仪表盘** + 趋势图（按月发表量、Top 期刊/机构）；**周报/简报一键导出**（PDF/Docx）；团队可共享订阅。
- **红线**：版权合规、明确数据留存策略；商用场景尤其不能碰灰色源。

### U5 · 实验室 PI / 课题组长
- **场景**：管几个方向，想要**本组主题的每日/每周简报**，还想知道"谁引用了我们的工作"。
- **核心诉求**：多主题订阅、团队共享、**引用追踪（cited-by）**、经费/趋势洞察、把好文献转给学生。
- **痛点**：精力有限、想委托但要可控、团队信息不同步。
- **对界面/内容**：**订阅 = 主题**的管理视图；简报可**转发/分配**给组员；引用我方的提醒单列。
- **红线**：团队共享时注意订阅内容（研究方向）的保密。

### U6 · 图书馆员 / 信息检索专员
- **场景**：替多个课题组建检索、教用户合法获取全文、维护机构订阅链路。
- **核心诉求**：**机构链接解析器（SFX / LinkResolver / OpenURL）**集成、为他人保存/分享检索式、合法获取教学、批量管理。
- **痛点**：用户乱用灰色源、机构全文链路没接好、检索式难复用。
- **对界面/内容**：**OpenURL/机构代理**可配置；检索式模板库；"如何合法拿到这篇"的引导步骤（机构 proxy→ILL→联系作者）。
- **红线**：强力倡导合法获取；这是该角色的职业立场，与产品红线天然一致。

### U7 · 政策研究 / 公共卫生 + 科普作者（合并：面向"非纯学术"受众）
- **场景**：跨学科找证据、要给**非专业受众**写通俗解读，怕误读预印本、要可信度信号。
- **核心诉求**：跨库 + 灰色文献（报告/指南）、按地区/人群、**通俗版总结**、可信度/证据等级提示、找综述优先。
- **痛点**：术语门槛、预印本风险、需要"给外行讲清楚"。
- **对界面/内容**：**"通俗摘要"开关**（面向公众语气）；**证据强度/是否同行评议**醒目标识；优先综述/指南的筛选。
- **红线**：通俗化不得歪曲；预印本必须标"未经同行评议"。

## 1.2 跨角色张力（必须在方案里调和，不能和稀泥）

1. **信噪比 vs 召回率**：U1/U7 要"少而准、AI 先帮判重要性"；U3 要"全、宁多勿漏、AI 别帮我筛"。
   → **方案**：AI 只做**排序与打标**（可关），**永不删条目**；U1 可开"按相关度/临床意义排序+折叠低分"，U3 可关排序、要全量 + 可复现检索式。同一引擎，两种呈现档（"晨读精炼" vs "系统检索"模式）。
2. **AI 判断 vs AI 不可越权**：U1 要"是否改变实践"判断；U3 要"AI 别替我下结论"。
   → **方案**：AI 输出永远是**带依据的建议**（标全文/摘要来源、附原文），措辞为"提示/可能"，**纳入排除按钮永远是人点**；系统综述模式下命中只进"待筛"。
3. **本地隐私 vs 推到手机/团队共享**：U1/U5 想要手机推送 + 团队共享；R-Sec/Audit 要隐私与留存可控。
   → **方案**：基线全本地；手机推送/共享为**显式可选**，用户自填通道/自托管，UI 明示数据流向。
4. **简单 vs 强大**：U1 要极简晨读；U4/U6 要多列/多订阅/仪表盘。
   → **方案**：默认极简，**渐进披露**（高级 facet、列配置、检索构造器收在"高级"里）。

## 2.2 数据源适配器（合法 + 守 ToS）

统一接口 `SourceAdapter.search(querySpec, since?) → SearchHit[]`，各适配器内部处理分页/速率/字段映射：

| 源 | 用途 | 关键约束 |
|---|---|---|
| **PubMed (E-utilities)** | 生物医学主库、MeSH、cited-by(elink) | 带 `tool`+`email`；有 key ≤10 req/s；ESearch→EFetch/ESummary |
| **Europe PMC** | 生物医学 + 全文 OA 链接 + 引用 | REST、OA 全文定位强 |
| **Crossref** | 跨学科元数据、DOI、被引 | polite pool(mailto)；版本/关系 |
| **OpenAlex** | 跨学科、机构/作者消歧、概念、OA 状态 | polite pool(mailto)；趋势统计利器 |
| **arXiv** | 物理/CS/数学 preprint | 官方 API，温柔轮询 |
| **bioRxiv/medRxiv** | 生命科学 preprint | 官方 API；preprint↔published 链接 |

**`QuerySpec`（结构化检索式，N-F2）**：
```ts
interface QuerySpec {
  groups: Array<{                  // 布尔组,组间 AND/OR
    op: 'AND'|'OR'|'NOT';
    terms: Array<{ field: 'title'|'abstract'|'tiab'|'author'|'journal'|'mesh'|'all'; value: string; explode?: boolean }>;
  }>;
  filters: { yearFrom?: number; yearTo?: number; sources?: string[]; types?: StudyType[];
             peerReviewedOnly?: boolean; openAccessOnly?: boolean; languages?: string[] };
  raw?: string;                    // 与结构化双向同步的原始检索式文本
}
```
每个适配器把 `QuerySpec` 翻译成自己的语法（PubMed 的 `[tiab]`/`[mesh]`、Crossref 的 query 参数……），并把 `raw` 暴露给 U3 存档。

## 2.3 归一化 + 去重 + 版本归并（N-F3）

- **归一化**：各源映射到统一 `Paper`（见 2.5），日期、作者、期刊缩写、类型标准化。
- **去重键**：① 有 DOI → DOI 规范化；② 无 DOI → `标题指纹(小写去标点 normalize) + 首作者姓 + 年`；③ 模糊兜底（标题 Jaccard/编辑距离阈值）。
- **版本归并**：preprint 与正式发表经 DOI 关系（Crossref `is-preprint-of` / bioRxiv published-DOI）合并为**一条带 versions[]**，UI 显示最新 + 可展开历史（U2/U3）。

## 2.4 调度与"每日推送"（会议 0 决定的 A/B/C 落地）

- **A 基线**：`Scheduler` 在 App 启动时对每个订阅跑 `since = max(last_run, 今天 0 点)` 的增量检索 → 命中入库 → 生成当日 `Digest` → 应用内"今日推送"展示 + （App 运行时）原生通知。**离线/睡眠醒来 = 自动补发**。
- **B 增强**：`auto-launch` 开机自启 + `node-cron` 按用户设的时间(含时区/安静时段, N-D6)定时跑；可作 headless sidecar 常驻发原生通知。
- **C 可选**：同一内核在常开机器/自托管跑，命中后经 `Notifier` 邮件/Telegram/Webhook 推送（N-D3）。
- **Kai 边界**：首次运行无 `last_run` → 只取"今天"避免回灌历史洪水；API 失败 → 该源标记 + 退避重试，**不让一个源拖垮整份简报**（部分成功）；命中 0 → 简报显式"今日无新命中"（非空白）。

## 2.6 检索 + Facet + 排序（N-U1/U2/U3/U4）

- **检索**：FTS5 over title/abstract + 结构化过滤（year/source/type/OA/lang/peer-reviewed）。
- **Facet 计数**：对当前结果集做分组聚合（`GROUP BY year/journal/source/type`）实时算每个 facet 选项的命中数（数据库式体验）。
- **排序**：相关度(FTS5 bm25 + 可选 embedding rerank) / 日期 / 被引 / 期刊分级。
- **命中理由(N-U4)**：FTS5 `snippet()` 高亮匹配词 + 标注命中字段。

## 2.7 AI 总结管线（你的核心诉求，N-S 全套）

```
候选(命中/全列表/单条)
   └─[选项: 总结来源]──────────────────────────────────────┐
        · 仅摘要      → 用 title+abstract 直接总结           │
        · 优先全文    → OaResolver→(合法)PdfFetcher→pdfjs 抽文本→分块 │
                         取到=基于全文; 取不到=回退摘要(标徽章) │
        · 不总结      → 跳过, 仅入库列出                      │
   └─[选项: 是否取 PDF] 是/否/仅OA可得 ── 复用红线: 仅合法OA  │
   └─[选项: 深度/风格] TL;DR / 结构化 / 临床 / 科普          │
   └─[选项: 语言] 中/英/双语                                  │
        ▼
   LLM(可插拔: Claude/OpenAI/Ollama) 按对应 prompt 模板生成
        ▼
   存 summaries(source_basis, depth, language, model) + 渲染
        ▼
   UI 显示: 【基于全文/基于摘要】徽章 + 原文链接 (反幻觉, N-S5)
```
- **R-ML**：结构化模板用固定 JSON schema（目的/方法/结果/结论/局限/样本量/研究类型）让输出稳定可渲染；临床模板加"是否可能改变实践 + 证据强度"字段（仅作**提示**）；科普模板换通俗语气并强制"是否同行评议"提示。
- **Audit/U3 红线**：总结**绝不输出"纳入/排除"建议**；纳入排除是 `paper_state.screening`，**只由用户按钮改**。
- **成本/隐私**：默认只总结"今日推送命中项"(N-S6)，全列表总结需手动触发并提示 token 成本；支持本地 Ollama 全程不出网。

## 2.8 UI 规格（落实会议 1 的"数据库风格 + 多筛选"）

- **三区布局**：左 **Facet 栏**（来源 / 年份滑块 / 期刊 / 研究类型 / OA·全文 / 语言 / 同行评议 / 证据等级，渐进披露高级项）｜中 **结果区**（搜索栏 + 检索构造器入口 + 排序 + 结果数 + 密度切换 + 文献卡列表）｜右/抽屉 **详情**（元数据 + 摘要 + AI 总结 + 获取入口 + 收藏/待筛/已读）。
- **顶部模式切换**：**探索** ↔ **今日推送**（晨读视图：按订阅分组、卡片化、每条 TL;DR + 取全文开关 + 加入待筛/收藏）。
- **文献卡**：左缘"**证据/时效脊**"（研究类型色 + 新近度）｜标题(衬线，期刊感)｜作者 · 期刊缩写·年·卷｜元数据 chips（DOI、被引、OA、语言、preprint/撤稿徽章）｜命中词高亮｜AI 一句话 TL;DR｜操作（取全文 / 收藏 / 待筛 / 详情）。
- **诚实状态**：preprint→"未经同行评议"黄标；撤稿→红标；Web/睡眠推送→"下次唤醒补发"提示；取全文失败→"打开 OA / 经机构访问"。
- **i18n**：中/英；术语（study type、journal quartile、MeSH）本地化。
- **可访问性**：键盘可达、facet 可键盘操作、`prefers-reduced-motion`、移动端晨读自适应。
- **设计语言（避免模板感）**：研究仪器气质——衬线标题（期刊排版感）+ 等宽元数据 + 克制的靛蓝强调色 + facet 计数像数据库；不用暖米色 + 大衬线 + 陶土色那套 AI 默认。

## 2.9 ADR（关键决议留痕）

- **ADR-1 仅合法 OA 自动下载**：复用 `OA_DENY_PATTERNS` + 解析器 + PDF 校验；主进程二次 DENY 纵深防御。
- **ADR-2 守 ToS**：每源限速器 + mailto/tool+email 署名 + 缓存 + 退避；不抓出版商 HTML 正文。
- **ADR-3 LLM 用户自持**：key 入钥匙串；支持本地 Ollama 全本地路径；总结成本对用户透明。
- **ADR-4 AI 不越权**：只排序/打标/总结（带依据徽章），**永不替做纳入/排除**；系统综述模式命中只进"待筛"。
- **ADR-5 本地优先**：零强制账号、离线可看已存；手机推送/团队共享为显式可选并明示数据流向。
- **ADR-6 部分成功**：多源检索任一源失败不阻断整体；简报标注哪个源缺数据。
